# 文档格式检查 Skills 项目背景与机会

## 1. 项目定位

- 项目代号：`doc-format-skills`
- 当前场景：高校论文 `.docx` / `.dotm` 格式检查
- 目标形态：一组可被 Claude Code agent 编排调用的 skills
- 核心目标：把“模板/规范 -> 规则元数据 -> 产物检查 -> 错误报告”拆成可复用的能力图，而不是一个写死流程的工具

## 2. 问题定义

高校论文格式要求通常分散在多个来源：

- 学校规范 PDF
- 学院提供的 Word 模板
- 历届合格样例
- 导师或教务补充说明

用户面临的问题不是只有“不会检查”，还包括：

- 规范信息分散，难以归一
- 模板样式和文字说明经常不一致
- 不同学校之间很难复用单一模板工具
- 同一个任务往往需要文本检索、样式查询、视觉核验、规则合并、结构验收等多步能力

这天然不是一个“大脚本”问题，而是一个 skill orchestration 问题。

## 3. 机会判断

如果把能力拆成细粒度 skills，agent 可以：

- 先读取结构事实
- 再查找规范文本线索
- 再查询样式默认值
- 再在冲突处做页面视觉核验
- 再合并为 spec
- 最后走 gate 验收闭环

这样系统的可扩展性会明显强于“固定两条主流程 + 固定脚本”的模式。

## 4. 产品方向

本项目仍然围绕两个用户目标展开：

1. **提取 spec**
2. **检查文档**

但系统内部不再把它们理解成两个孤立成品功能，而是理解成两个 workflow，它们共享底层 skill graph。

## 5. 推荐能力分层

### Primitive Skills

- `parse-word` - 解析 Word 文档，支持中英文字体分离、页眉页脚提取
- `query-word-text` - 按关键词检索文本
- `query-word-style` - 查询样式定义
- `render-word-page` - 渲染页面截图
- `read-text` - 读取文本文件

### Analysis Skills

- `infer-spec-fragment` - 推断局部规则片段
- `merge-spec-fragments` - 合并多个 spec fragments
- `compare-doc-layout` - 比对文档布局差异

### Gate Skills

- `validate-spec` - 校验 spec.json 结构（支持 --strict 严格模式）
- `validate-report` - 校验报告结构

### Workflow Skills

- `extract-spec` - 从模板提取 spec
- `check-thesis` - 检查论文格式，输出 skipped_rules
- `agent-check-report` - 智能检查跳过的规则（新增）
- `compare-docs` - 比对文档

## 6. 首阶段边界

首阶段只聚焦：

- 输入主格式：`.docx` / `.dotm`
- 主场景：高校论文 / 毕业设计
- 主输出：`spec.json`、`report.json`、`report.md`

暂不优先做：

- 自动修复
- PDF 主检查流
- 复杂公式、文本框、图形对象的细粒度校验

## 7. 成功标准

- agent 能围绕真实模板文档调用多个 skills 生成可用 `spec.json`
- gate 能发现 spec 缺漏并驱动补偿提取
- `spec.json` 能驱动待检查文档输出可解释报告
- 新增一个原子能力时，不需要改写整套系统结构
- 混合检查模式：Python 检查已知 selector，Agent 检查语义化 selector
