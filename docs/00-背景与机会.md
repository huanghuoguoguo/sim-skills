# 文档格式检查 Skills 项目背景与机会

## 1. 文档信息

- 项目代号：`doc-format-skills`
- 调研日期：2026-03-31
- 目标：为“模板/规范 -> 规则元数据 -> 产物检查/比对 -> 错误报告”产品化方案建立背景共识
- 当前落地场景：高校论文 `.docx` 格式检查
- 长期方向：通用 `artifact comparison/compliance engine`

## 2. 问题定义

高校论文、毕业设计报告、课程论文、期刊投稿稿件都存在大量格式合规要求。这些要求通常分散在以下材料中：

- 学校论文撰写规范 PDF
- 学院提供的 Word 模板
- 历届合格样例
- 导师或教务补充说明

用户常见痛点：

- 不知道要求到底是什么，规范文本分散且含糊
- 即使知道要求，也很难逐条在 Word 中手工核对
- 学校之间规则差异大，通用模板工具无法直接复用
- 检查时通常要导师、同学、辅导员或人工服务反复来回
- 最终错误集中在细节层面，如行距、段前后距、标题层级、页码、题注、表注、参考文献、图片布局

本项目要解决的不是“内容写作”，而是“规则的结构化提取、产物归一化理解与自动合规/差异检查”。

进一步抽象后，项目面对的是一类更通用的问题：

- 有一个“规范、模板、样例、目标状态”
- 有一个“待交付的实际产物”
- 系统需要判断两者是否一致，或者实际产物是否符合规范

在当前阶段，这个问题先收敛为“学校论文模板 vs 学生 `.docx` 论文”。

## 3. 目标用户

- 本科生、硕士生、博士生
- 指导老师、教务人员、论文格式审查人员
- 提供论文辅导服务的机构
- 需要定制内部文档格式规则的组织
- 长期还可覆盖需要“规范 vs 产物”校验的团队，如内容团队、品牌团队、设计审查团队、交付审核团队

## 4. 竞品与替代方案

### 4.1 格子达格式检测

公开信息显示，格子达已提供“按国标与学校要求检测”的格式检测能力，检查项覆盖：

- 文档主体完整性
- 题目、封面、摘要、关键词、目录
- 各级标题与正文的字体样式、段落样式
- 图片、表格、分页、页眉页脚、页边距
- 参考文献格式检测

它证明了“高校论文格式检测”是被验证过的需求，但其核心更像是平台预置规则库或校方集成能力。

### 4.2 维普 OA 稿件服务

维普提供稿件格式检测服务，面向学术投稿流程，强调：

- 文章结构规范性检测
- 与查重、敏感词、错别字等服务并列
- 作者上传稿件后生成格式分析报告

它证明“格式检测”可以独立成为付费能力，但更偏期刊稿件场景，不一定覆盖学校模板级别的细粒度差异。

### 4.3 FreeFormat

FreeFormat 支持：

- 上传 `.docx`
- 输出格式检查分数与问题清单
- 将文档套用到现有模板上

其公开说明明确表示目前“不支持从上传指南中创建模板”。这正说明市场上已有“检查 + 套模板”产品，但“用户上传模板并自动抽取规则”仍是空缺点。

### 4.4 PerfectIt / Acrolinx 类工具

这类工具擅长：

- 根据 style guide 检查文风、术语、拼写、标点、列表规范
- 支持组织自定义规则

但它们主要解决的是语言与风格一致性，不是 Word 版式与学位论文格式细则。

## 5. 竞品结论

现有方案大致分为两类：

- 预置规则库检查型：已有固定学校或固定规范，用户直接上传文档检测
- 风格指南治理型：允许导入 style guide，再检查文本是否符合规则

而我们当前切入的是第三类：

- 用户上传模板或规范材料
- AI 抽取为结构化、可执行的格式元数据
- 再用程序 + AI 对用户文档进行逐条合规检查

这是比“通用格式检测”更强的产品定义。

长期看，它还可以继续抽象为第四类：

- 任意两份可解析产物的统一比对与合规框架
- 输入不再局限于论文 `.docx`
- 既支持“规范校验”，也支持“样例对齐”和“产物差异分析”

## 6. 我们为什么要做

### 6.1 差异化明确

如果只做“论文格式检测”，会直接进入已有玩家赛道；如果把重点放在“模板自动理解与规则抽取”，就获得了更强的适配能力。

### 6.2 AI 在这里确实有增益

纯规则引擎很难直接消费学校规范文本，因为规范常常是：

- 非结构化自然语言
- 不完整
- 存在例外
- 依赖上下文
- 分散在多个文件

AI 可以承担：

- 从模板、样例、PDF 规范中提取候选规则
- 将模糊规范转成结构化字段
- 识别冲突、缺失和不确定项
- 将程序发现的问题翻译成用户可理解的说明

### 6.3 程序化能力仍然是主引擎

真正决定产品可用性的不是“AI 会说”，而是“可稳定复现地检查具体产物”。因此 AI 只负责高不确定性的部分，检查主链仍由程序完成。

### 6.4 高价值切入点

论文格式是典型的高频低创造性任务：

- 用户愿意为省时间付费
- 规则重复、可积累
- 可沉淀学校/学院模板资产
- 容易形成规则库和机构版产品

它也是理想的切入场景，因为：

- `.docx` 结构化程度高，便于程序化解析
- 错误规则明确，便于建立规则模型
- 用户反馈闭环短，便于快速验证
- 成功后可向其他 artifact 类型平移

## 7. 产品定位

项目定位不是通用写作助手，而是：

“以高校论文为切入场景的 artifact comparison/compliance engine”

其核心能力包括：

- 模板/规范理解
- 规则元数据生成
- 产物结构与属性解析
- 规则匹配、差异计算与错误定位
- 报告生成与部分自动修复

当前的具体产品表达仍然是：

- 面向高校论文与强格式文档的规则抽取与格式合规检查引擎

长期的抽象表达是：

- 面向多种 artifact 的统一比对与合规校验引擎

## 8. 核心抽象

为了避免后续推倒重来，系统应从一开始使用以下核心概念：

- `artifact`：任意输入对象，如 `.docx`、PDF、HTML、截图、样例文件
- `normalized representation`：归一化后的中间表示
- `fact`：从 artifact 中提取出的可比属性
- `spec`：规则、目标状态或参考样例的结构化表达
- `selector`：指定检查对象的定位方式
- `comparator`：比较事实与规则、或比较两个 artifact 的模块
- `issue`：差异、违规项或待确认项

这组抽象允许系统支持三类任务：

- `compliance check`：产物是否符合规范
- `artifact diff`：A 和 B 有哪些差异
- `reference alignment`：产物是否与样例或模板保持一致

## 9. 范围边界

首阶段建议只做：

- 输入主格式：`.docx`
- 主场景：高校论文/毕业设计/课程报告
- 主能力：检查，不先追求全自动修复
- 主模式：`spec -> artifact` 的合规检查

暂不优先做：

- PDF 作为主检查对象
- WPS 私有格式的深支持
- 复杂公式、文本框、艺术字的细粒度渲染一致性
- 完整参考文献内容正确性判断
- 通用视觉 diff
- 任意格式两两混合比对

## 10. 成功标准

从产品验证角度，第一阶段成功不看“功能多”，而看以下指标：

- 能否稳定从模板或规范中生成一份可执行规则 JSON
- 能否在真实 `.docx` 上定位主要格式问题
- 错误报告是否对用户有可操作性
- 新学校模板接入是否足够低成本
- 是否能逐步沉淀为模板库与规则库

从架构正确性角度，还应额外满足：

- 核心模块命名与接口不把能力写死在 `.docx`
- 新增 artifact 类型时，以新增 adapter 为主，而不是重写引擎
- 同一套 `selector / comparator / issue` 模型可复用到后续场景
