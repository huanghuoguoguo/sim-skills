# 文档格式检查 Skills 实施方案

## 1. 实施目标

围绕完整产品目标，建设一个可扩展的 artifact comparison/compliance engine，并为后续 MVP 和产品化迭代提供工程基础。

当前实施策略是：

- 内核做通用
- adapter 从 `.docx` 开始
- 业务场景先聚焦高校论文

## 2. 工程范围

完整实施包含四个子系统：

- `ingestion`：模板、规范、样例、用户产物接入
- `adapters`：不同 artifact 到统一 IR 的转换
- `fact-extraction`：从 IR 中提取可比较事实
- `spec-builder`：规则候选抽取、归一化、审核、发布
- `checker`：规则执行、差异计算、结果输出
- `reporting`：用户报告、管理后台、调试工具

## 3. 技术建议

### 3.1 语言与框架

- 后端：Python
- API：FastAPI
- 任务队列：Celery 或 RQ
- 数据库：PostgreSQL
- 对象存储：本地文件系统起步，后续可切 S3 兼容存储

### 3.2 Artifact 处理

- `.docx` 解析：优先基于 OOXML 直接解析，必要时辅以 `python-docx`
- PDF 文本抽取：`pdfplumber` 或 `pymupdf`
- HTML 解析：后续可引入 `beautifulsoup4` 或 DOM 解析器
- 图像/截图理解：后续可引入 OCR 与视觉结构分析
- JSON schema 校验：`pydantic`

### 3.3 AI 集成

- 模型调用封装成独立服务
- 所有模型输出必须走 schema 校验
- 模型参与点仅限规则抽取、归一化、解释生成

## 5. 分阶段开发

### 阶段 1：通用内核

目标：

- 定义 `Artifact IR`
- 定义 `Fact` 模型
- 定义 spec schema
- 实现最小 comparator / rule engine

产出：

- 通用内核接口
- `.docx` 适配器骨架
- rule matcher
- 几类基础规则检查

### 阶段 2：docx 垂直打通

目标：

- 解析 `.docx`
- 从模板和规范中生成候选规则
- 引入 AI 归一化流程
- 增加规则草稿与审核机制

产出：

- spec draft 生成流水线
- 冲突检测与单位归一化
- spec 存储与版本管理

### 阶段 3：检查与比对报告

目标：

- 让用户上传 artifact 并运行检查或比对
- 输出 JSON 和可读报告

产出：

- check API
- compare API
- HTML/Markdown 报告
- 基础分数与错误分级

### 阶段 4：产品化能力

目标：

- 多模板管理
- 后台审核
- 可视化问题浏览

产出：

- spec 管理界面
- 任务状态跟踪
- 结果查询与导出

## 6. 核心开发任务拆分

### 6.1 IR 与 Fact 设计

- 定义通用 `Artifact IR`
- 为文档类定义段落、run、表格、图片、节、页边距、页眉页脚结构
- 定义从 IR 投影出的 `Fact` 模型
- 约束字段单位和枚举值
- 建立 fixture 文档和 snapshot

### 6.2 Adapter 实现

先实现 `.docx` adapter：

- 读取 `document.xml`
- 读取 `styles.xml`
- 读取 `numbering.xml`
- 解析节属性与页面设置
- 建立样式继承解析

后续按需补充：

- PDF/text adapter
- HTML adapter
- image/layout adapter

### 6.3 Spec Schema

- 设计 selector 语法
- 设计 property 语法
- 定义 rule / exception / source 数据结构
- 建立 schema version 机制

### 6.4 Comparator / Rule Engine

- selector 匹配器
- property comparator
- artifact diff comparator
- severity 计算
- issue formatter

### 6.5 Spec Builder

- 模板事实抽取器
- 规范文本片段抽取器
- AI 归一化提示词与输出 schema
- 冲突与缺失项分析器

### 6.6 Reporting

- JSON 结果模型
- Markdown/HTML 渲染
- 按区块与规则汇总
- 为后续多 artifact 类型保留统一报告骨架

## 7. 数据模型建议

### 7.1 主要表

- `files`
- `specs`
- `spec_versions`
- `spec_sources`
- `check_jobs`
- `compare_jobs`
- `check_results`
- `issues`

### 7.2 核心关系

- 一个 `spec` 有多个 `spec_version`
- 一个 `spec_version` 关联多个源文件
- 一个 `check_job` 绑定一个 `spec_version`
- 一个 `check_job` 产出多个 `issue`
- 一个 `compare_job` 绑定两个 artifact 输入

## 9. 测试策略

### 9.1 单元测试

- selector 匹配
- property 比较
- schema 校验
- adapter 解析函数
- comparator 逻辑

### 9.2 集成测试

- 模板 -> spec draft
- 文档 -> issues
- artifact A + artifact B -> diffs
- report 输出稳定性

### 9.3 回归样本

建立固定样本集：

- 正确文档

## 10. 非功能要求

- 结果稳定，可重复
- 所有模型输出可审计
- 报告生成耗时可控
- 原始文档默认保密存储

## 12. 主要风险

- Word 解析细节复杂，尤其是样式继承和节属性
- 不同学校模板存在例外规则
- AI 抽取规则可能不稳定
- 页面级位置难以低成本精确还原
- 为未来扩展而过度设计，拖慢当前 MVP

## 13. 风险控制

- 先做高价值字段，避免一次覆盖全部 Word 能力
- 所有规则抽取结果必须可人工确认
- 用 fixture 驱动 parser 和 checker 开发
- 把 selector 和 rule engine 做成完全可测模块
- 保持核心通用，但只实现一条 `docx` 垂直链路
