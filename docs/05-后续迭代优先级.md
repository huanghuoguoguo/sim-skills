# 文档格式检查 Skills 后续迭代特性优先级

## 1. 目标

在 MVP 跑通后，按“用户价值 / 技术依赖 / 交付成本”排序，规划后续迭代方向。

## 2. 优先级原则

- 先提高检查准确性，再扩展功能面
- 先做高频规则，再做低频边角能力
- 先做可解释的自动化，再做侵入式自动修复
- 先扩展可复用内核，再扩展新 artifact 类型

## 3. P0：MVP 必须打稳

- 提高正文、标题、题注规则识别准确率
- 完善 `pending_confirmations` 机制
- 增加不同学校模板的回归样本
- 改善 issue 定位与提示文案

## 4. P1：最值得做的下一批能力

- 支持摘要、目录、参考文献区块识别
- 支持更多段落属性与编号规则
- 支持页眉页脚和页码基础检查
- 支持规则审核与发布流程
- 支持模板库和规则版本管理
- 把 `artifact / fact / comparator / issue` 内核接口稳定下来

价值：

- 这批能力能把产品从“演示工具”推到“可交付工具”

## 5. P2：形成产品差异化的能力

- 从 PDF 规范中抽取规则
- 多源融合建模：模板 + 指南 + 样例联合抽取
- 冲突规则自动解释
- 问题严重级别智能排序
- 机构版模板管理与复用
- 引入 `artifact A vs artifact B` 差异比对模式

价值：

- 这批能力决定“是不是比普通规则检查器更强”

## 6. P3：提升体验但不应过早投入的能力

- 自动修复部分格式问题
- 可视化逐项对比界面
- 批量上传与批量检查
- 协作审查与评论流
- 结果导出为 Word 批注或修订建议
- 增加 HTML、截图、设计稿等新 adapter

价值：

- 体验提升明显，但依赖底层规则稳定后再做

## 7. P4：长期探索项

- PDF 视觉版式检查
- WPS 深度兼容
- 复杂公式和图形对象检查
- 参考文献内容级规范检查
- 多语种论文规则支持
- 跨模态 artifact 比对，如“规范截图 vs 实际页面截图”

## 8. 推荐迭代顺序

1. 做准：提高 parser、selector、rule engine 稳定性
2. 做宽：覆盖更多论文区块与规则类型
3. 做深：加入多源规则抽取、审核工作流和 artifact diff
4. 做顺：加入自动修复和更好的交互体验
5. 做广：扩展到更多文档类型和机构场景

## 9. 每阶段判断标准

### 从 P0 进入 P1

- 样本回归稳定
- 核心误报明显下降
- 新模板接入成本可接受

### 从 P1 进入 P2

- 用户开始提出模板管理和规则复用需求
- 单模板检查已不是主要瓶颈

### 从 P2 进入 P3

- 检查结果可信度足够高
- 用户明确希望“一键修复”而不是只看报告

## 10. 当前建议

短期不建议过早投入自动修复、复杂前端和 PDF 视觉检查。当前最该做的是把：

- `.docx` 解析
- artifact 核心抽象
- 规则 schema
- 规则抽取
- 规则执行
- 报告表达

这五件事先打稳。
