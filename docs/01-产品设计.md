# 文档格式检查 Skills 产品设计

## 1. 设计目标

设计一组面向 Claude Code 的细粒度 skills，让 agent 能围绕文档格式任务自主编排 workflow，而不是依赖少数写死流程的大技能。

第一条落地链路仍然是论文 `.docx` 场景，但产品形态从一开始就按 **skill graph** 设计，而不是按“单体工具”设计。

系统当前关注两类用户目标：

1. 从模板/成品文档中提取可复用的 spec
2. 用已有 spec 检查待提交文档并输出问题报告

## 2. 核心原则

- skill-first：能力优先拆成 skill，再由 workflow 组织
- programmatic-first：能用程序稳定拿到的事实，不交给模型猜
- agentic-orchestration：workflow skill 负责决策和编排，不负责吞掉所有逻辑
- explicit-artifacts：每一步都尽量产出明确 artifact，便于下一步消费
- closed-loop：输出 spec/report 前必须经过 gate skill 验收

## 3. 产品边界

### 3.1 输入

- 模板 `.docx` / `.dotm`
- 成品 `.docx`
- 待检查 `.docx`
- 规范文本 `.txt` / `.md` / 从文档中抽出的要求段落
- 已有 `spec.json`

### 3.2 输出

- `DocumentIR`
- `spec fragment`
- `spec.json`
- `report.json`
- `report.md`

### 3.3 首阶段不优先支持

- 自动修复文档
- PDF 作为主检查对象
- 页面级精确排版判断
- 复杂公式、文本框、图形对象的精细校验

## 4. 总体架构

### 4.1 Skill Graph

```
用户目标
  ↓
workflow skill
  ↓
primitive skills → analysis skills → gate skills
  ↓
artifact 持续沉淀
```

不是：

```
用户目标 → 一个大 skill → 一个大脚本 → 最终结果
```

### 4.2 四层模型

#### Primitive Skills

负责读取和暴露底层事实，不做业务判断。

候选 skills：

- `parse-word` - 支持中英文字体分离 (font_family_east_asia/font_family_ascii)、页眉页脚提取
- `query-word-text`
- `query-word-style`
- `render-word-page`
- `read-text`

#### Analysis Skills

负责把原始事实变成可用于规则推断和报告生成的中间结果。

候选 skills：

- `infer-section-types`
- `infer-spec-fragment`
- `merge-spec-fragments`
- `compare-doc-layout`

#### Gate Skills

负责把输出卡在结构正确、覆盖足够、可继续执行的状态。

候选 skills：

- `validate-spec` - 支持 --strict 严格模式，默认宽松验证
- `validate-report`

#### Workflow Skills

面向用户目标，负责决策、编排、分支、回退和闭环。

候选 skills：

- `extract-spec`
- `check-thesis` - 输出 skipped_rules 列表
- `agent-check-report` - 智能检查跳过的规则
- `compare-docs`

## 5. Skill 契约

每个 skill 都应该写清楚以下契约，而不是只写“执行哪个脚本”：

### 5.1 输入契约

- 接收什么 artifact
- 接收什么路径
- 对输入完整性的最低要求是什么

### 5.2 输出契约

- 输出什么 artifact
- 输出是否稳定可复用
- 输出是否需要落盘

### 5.3 调用时机

- 什么时候应该调用这个 skill
- 什么时候不应该调用这个 skill
- 调用失败时由谁兜底

### 5.4 下游关系

- 典型下一跳 skill 是谁
- 结果不确定时进入哪个 gate skill
- gate 不通过时回退到哪个上游 skill

## 6. Skill 拆分准则

### 6.1 总原则

不是“粒度越细越好”，而是：

**在可复用、可组合、可替换、可验收的前提下，尽量细。**

如果一个能力拆细后只会增加编排负担，而不会带来复用、替换、验收或回退价值，就不应该强行升格成独立 skill。

### 6.2 什么时候应该拆成独立 skill

满足以下任意多项时，优先考虑拆成独立 skill：

- 会被多个 workflow 复用
- 有明确输入和输出 artifact
- 值得被单独替换实现
- 值得被单独验收
- 失败后需要单独回退或补偿
- 未来可能接入不同模型能力或不同工具链

典型例子：

- `parse-word`
- `query-word-text`
- `query-word-style`
- `render-word-page`
- `validate-spec`

### 6.3 什么时候不应该拆成独立 skill

出现以下情况时，更适合保留为 workflow 内部步骤、analysis 逻辑或脚本函数：

- 只服务于单一 skill 的内部细节
- 没有稳定 artifact，只是中间临时变量
- 没有独立复用价值
- 拆出来后只会形成“每个函数一个 skill”的噪音

原则上，不把普通函数调用直接等同于 skill。

### 6.4 Skill、Script、Agent 的职责分工

#### Agent

负责：

- 决定调用顺序
- 判断是否继续取证
- 判断是否需要视觉核验
- 判断是否进入 gate
- 处理失败回退和补偿提取
- 基于结果决定何时结束

#### Skill

负责：

- 暴露稳定能力或稳定 workflow
- 定义输入输出契约
- 定义适用场景和下一跳关系

#### Script

负责：

- 执行确定性动作
- 输出稳定结构
- 承担易错、重复、适合程序化的步骤

经验上：

- 容易反复重写的逻辑，应下沉到脚本
- 需要 agent 判断和分支的逻辑，应保留在 workflow skill

### 6.5 Workflow Skill 与 Capability Skill

本项目允许两类 skill 并存：

#### Workflow Skill

特点：

- 面向用户目标
- 会编排多个下层 skills
- 包含分支、回退、验收和终止条件

例如：

- `extract-spec`
- `check-thesis`

#### Capability Skill

特点：

- 面向单一稳定能力
- 不关心完整业务流程
- 更强调明确输入输出

例如：

- `parse-word`
- `query-word-style`
- `validate-report`

### 6.6 视觉能力接入原则

视觉能力是增强通道，不是默认唯一依据。

推荐优先级：

1. 程序化事实
2. 文本规范线索
3. 视觉核验
4. gate 验收

也就是说：

- 能从 XML、样式、结构中确定的事实，优先走程序化路径
- 当文本说明、样式定义和页面效果冲突时，再调用视觉能力裁决
- 如果当前驱动模型不支持视觉，应允许 workflow 自动退回纯程序化路径

因此，视觉能力应被建模为可选 capability skill，例如：

- `render-word-page`
- `inspect-page-layout`

### 6.7 Artifact 优先于函数边界

本项目拆 skill 时，优先按 artifact 边界切，而不是按代码函数切。

更稳定的切法是：

- `docx -> DocumentIR`
- `DocumentIR -> spec fragment`
- `spec fragment -> spec.json`
- `spec.json + DocumentIR -> report.json`

而不是：

- 一个函数一个 skill
- 一个判断分支一个 skill

### 6.8 Gate 必须独立

验收能力不要藏在 workflow 内部。

凡是下面这些动作，优先独立成 gate skill：

- 结构完整性校验
- 覆盖范围校验
- 报告质量校验

原因是：

- gate 需要独立演进
- gate 需要被多个 workflow 复用
- gate 失败时要清楚地驱动上游补偿

## 7. 关键 artifact 设计

### 7.1 DocumentIR

由 primitive skills 产出，描述文档结构事实。

典型内容：

- 段落（支持 font_family_east_asia/font_family_ascii 分离）
- 样式
- layout
- 页边距
- 文本内容
- 样式默认值
- 页眉页脚（headers/footers）

### 7.2 Spec Fragment

由 analysis skills 产出，表示某个局部区域或某类 selector 的规则候选。

例如：

- `body.paragraph` 候选规则
- `body.heading.level1` 候选规则
- `abstract.body` 候选规则
- `layout.page_margins` 候选规则

### 7.3 Spec JSON

由 workflow skill 汇总多个 fragment 后生成，再经过 gate skill 验收。

### 7.4 Report JSON / Markdown

由检查类 workflow 生成，要求能追溯到：

- 命中的 rule
- 对比属性
- 期望值
- 实际值
- 定位信息
- 置信说明或不确定说明

支持 `skipped_rules` 列表，供 `agent-check-report` 消费。

## 8. 用户路径

### 8.1 提取 Spec

推荐 workflow：

1. `extract-spec` 接收模板或成品文档
2. 调用 `parse-word` 获取全量结构
3. 调用 `query-word-text`、`query-word-style` 查找规范线索
4. 必要时调用 `render-word-page` 做视觉校验
5. 调用 `infer-spec-fragment` 形成局部规则候选
6. 调用 `merge-spec-fragments` 合并成 `spec.json`
7. 调用 `validate-spec` 验收
8. 若 gate 失败，回退到相应上游 skill 继续补取

### 8.2 检查文档

推荐 workflow：

1. `check-thesis` 接收待检查文档与 `spec.json`
2. 调用 `parse-word` 获取结构事实
3. Python 检查支持的 selector，输出 `skipped_rules` 列表
4. 调用 `agent-check-report` 检查跳过的规则（语义匹配）
5. 生成 `report.json`
6. 调用 `validate-report`
7. 输出 `report.md`

### 8.3 比对文档

推荐 workflow：

1. `compare-docs` 接收参考文档与目标文档
2. 调用 `parse-word` 解析两份文档
3. 调用布局/样式比对类 analysis skill 输出差异
4. 视需要把差异再转成 spec fragment 或问题报告

## 9. 现有 skills 的定位调整

当前仓库里的 skills 可以先这样理解：

- `parse-word` / `query-word-text` / `query-word-style` / `render-word-page`
  已落地的 primitive skills
- `validate-spec` / `validate-report`
  已落地的 gate skills
- `extract-spec`
  当前是 workflow skill
- `check-thesis`
  当前是 workflow skill（支持 skipped_rules）
- `agent-check-report`
  当前是 workflow skill（语义匹配 skipped_rules）
- `infer-spec-fragment` / `merge-spec-fragments`
  当前先以 analysis skill 定义存在，后续还可以继续补脚本实现
- `compare-docs`
  当前是分析型/工作流混合 skill
- `read-text`
  通用辅助 skill
- `word`
  兼容聚合层，保留旧脚本路径

## 10. 风险与应对

### 9.1 过早把 workflow 写死在代码里

应对：

- workflow 只表达顺序、分支和回退
- 事实提取、局部推断、验收尽量拆成独立 skill

### 9.2 skill 粒度过细导致调用成本高

应对：

- 先按 artifact 边界切，不按函数数量切
- 只把可复用、可组合、可替换的能力升格为独立 skill

### 9.3 gate skill 写得过死

应对：

- 结构校验与覆盖校验分离
- 允许按场景定义不同 coverage profile
- 不把所有论文模板都绑在同一组固定 selector 上
