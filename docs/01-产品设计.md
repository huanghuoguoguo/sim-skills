# 产品设计

## 1. 设计目标

让 Claude Code（Agent）能够自主完成论文格式规则提取和文档格式检查，通过调用工具脚本高效处理确定性任务，通过自身推理处理语义性任务。

**核心原则：**

- **架构简单**：只有两个角色——Skill（指导 Agent）和工具脚本（Agent 的手和眼）
- **实现高效**：工具脚本可以内部复杂，但对外接口简单、可单步执行
- **可观测**：每一步做了什么、结果是什么，用户能看到
- **Agent 自主**：Skill 不规定步骤，只告诉 Agent 目标和质量标准，Agent 自己规划

## 2. 架构

```
用户请求
  → Claude Code 读取对应 Skill（SKILL.md）
  → Agent 自主规划
  → 调用 sim_docs CLI / Python API 获取事实 / 执行检查
  → Agent 综合推理
  → 输出产物
```

核心组件：

### Skill（SKILL.md）

- 指导 Agent 怎么思考、规划、判断质量
- 不规定具体步骤，Agent 自主编排
- 定义目标、输入输出、质量标准

### sim_docs 服务层

- **CLI**: `python3 -m sim_docs <command>` — 一shot 模式，适合简单调用
- **Python API**: `DocumentService` 类 — 可复用，内置 LRU 缓存
- 无业务判断：只提取事实或执行比对，不做"合不合格"的决定
- 上下游共享：同一批命令服务两条路径

## 3. 工具脚本

工具脚本已整合为统一的 `sim_docs` 服务层（CLI + Python API）：

| CLI 命令 | 职责 | 说明 |
|----------|------|------|
| `python3 -m sim_docs parse` | 解析 docx → 结构化事实 | 段落、样式、布局、页眉页脚、中英文字体分离 |
| `python3 -m sim_docs query-text` | 按关键词检索段落 | 用于从文档中找格式说明文字 |
| `python3 -m sim_docs query-style` | 查询样式属性（含继承链解析） | 返回最终解析后的属性值，不是"未设置" |
| `python3 -m sim_docs render` | 渲染页面为图片 | 给 Agent 视觉能力用，冲突仲裁时调用 |
| `python3 -m sim_docs check` | 批量格式检查 | Agent 传入检查指令，Python 批量执行，输出 pass/fail |
| `python3 -m sim_docs stats` | 段落统计 | 按条件筛段落并统计属性分布 |
| `python3 -m sim_docs validate` | XML 结构校验 | XSD schema 校验 + 自动修复 |
| `python3 -m sim_docs inspect` | 解包查看原始 XML | 用于调试边缘情况 |

工具脚本的核心要求：

1. **query_style 必须解析完整继承链**（直接格式 > 段落样式 > 父样式 > 文档默认），这是地基
2. **check 接受结构化检查指令**，Agent 从 spec.md 翻译规则为指令，Python 高效执行
3. 每个命令独立可运行，不依赖 Agent 上下文
4. **DocumentService 提供内置 LRU 缓存**，避免重复解析

### Skill 不做什么

- 不规定调用顺序（Agent 自己规划）
- 不做程序性门禁（由 Agent 自行判断质量）
- 不定义 artifact schema（产物是自然语言为主）

## 5. 上游：提取规则

### 输入

用户提供一批文件，可能包括：

- 学校模板（`.docx` / `.dotm`）——样式就是元数据
- 学长成品论文——格式示例
- 说明文档——直接写"正文宋体小四"这样的自然语言规格

### Agent 工作方式

Agent 读取 `extract-spec` Skill 后自主规划：

- 识别每个文件的类型（模板？成品？说明？）
- 对模板/成品：调用 `sim_docs parse` + `sim_docs query-style` 提取格式事实
- 对说明文档：调用 `sim_docs query-text` 或直接阅读，提取规则描述
- 必要时调用 `sim_docs render` 做视觉核验
- 综合所有来源，归纳为一份自然语言规则文档

### 输出

`spec.md` —— 自然语言格式规则，用户可读、可改。

### 质量评估

Agent 调用 `evaluate-spec` 自评：

- 常见格式要素是否覆盖（页边距、字体、字号、行距、标题层级…）
- 表述是否明确可执行（不是"字体要合适"而是"正文中文宋体小四"）
- 有无矛盾或明显遗漏

评估结果告知用户，用户精校后定稿。

## 6. 下游：检查文档

### 输入

- 用户的论文 `.docx`
- 定稿的 `spec.md`

### Agent 工作方式

Agent 读取 `check-thesis` Skill 后自主规划：

1. 读 `spec.md`，理解每条规则
2. 调用 `sim_docs parse` 解析论文
3. **确定性规则**（字体、字号、行距、边距、缩进、对齐…）→ 调用 `sim_docs check`，批量扫描
4. **语义性规则**（摘要内容、参考文献格式、目录结构…）→ Agent 用 `sim_docs query-text` + 推理判断
5. 必要时调用 `sim_docs render` 做视觉复核
6. 汇总输出检查报告

### 输出

Markdown 检查报告，每条结果包含：

- 规则描述
- 检查结果（通过/不通过）
- 期望值 vs 实际值
- 定位信息（哪一段、哪一页）
- 来源标注（Python 检查 / Agent 判断）

### 完整性保障

检查完成后，Agent 对照 spec.md **逐条回溯**，确认每条规则都被检查到，缺的补上。

## 7. 设计决策记录

### 为什么不用 spec.json？

- 上游产物给人看、给人改，自然语言最合适
- 下游消费者是 Agent（LLM），天然能读自然语言
- 去掉 spec.json 就去掉了 schema 定义、fragment 推断/合并、结构验证整条链

### 为什么用 Python + Agent 混合检查？

- 纯 Agent 检查：慢、贵、结果不稳定
- 纯 Python 检查：需要结构化规则输入，且无法处理语义性规则
- 混合：Python 批量扫描确定性规则（~80%），Agent 处理语义性规则（~20%），兼顾效率和灵活性

### 难点在哪？

1. **样式继承解析**：Word 格式层层继承，工具必须返回最终解析值——这是地基
2. **语义元素识别**：哪些段落是摘要、正文、参考文献？Agent 靠位置和上下文推理
3. **检查完整性**：Agent 容易漏查规则，需要 checklist 回溯机制
