# 文档格式检查 Skills 产品设计

## 1. 设计目标

设计一个围绕“统一 artifact 比对与合规检查”抽象的系统，并用 `.docx` 论文场景作为第一条落地链路。

系统支持三条主流程：

1. 模板/规范材料 -> 格式规则元数据
2. 参考产物 A + 待检查产物 B -> 差异报告
3. 规则 spec + 待检查产物 -> 合规检查报告

系统输出不是模糊建议，而是结构化、可解释、可复核的检查结果。

## 2. 核心原则

- 程序优先：能用程序判断的，不交给模型猜
- AI 补全：只让 AI 处理非结构化规范和模糊语义
- 可解释：每条错误都必须能说明“要求值、当前值、定位、依据”
- 可扩展：规则 schema 可逐步增加，不锁死在论文场景
- 可沉淀：模板规则应能复用、版本化和审核
- 抽象稳定：核心模型使用通用概念，不把引擎写死在单一文件格式

## 3. 系统能力边界

### 3.1 输入

- 参考 artifact：模板、样例、目标产物
- 规范 artifact：PDF、文本、表格、说明文档
- 待检查 artifact：用户实际提交内容

当前首批支持：

- 学校 Word 模板 `.docx`
- 学校格式说明 PDF 或文本
- 样例论文 `.docx`
- 用户待检查论文 `.docx`

### 3.2 输出

- 规则元数据 JSON
- 规则抽取置信度与待人工确认项
- 比对/检查报告 JSON
- 面向用户的可读报告 Markdown / HTML

## 4. 总体架构

```text
模板/规范输入
  -> Artifact Adapter
  -> 归一化表示层
  -> 规则候选抽取层
  -> AI 规则归一化层
  -> 规则审核/存储层
  -> 规则 JSON

参考产物/待检查产物输入
  -> Artifact Adapter
  -> 归一化表示层
  -> Fact Extraction
  -> Comparator / Rule Engine
  -> 问题聚合与分级
  -> 报告生成层
  -> 检查结果
```

## 5. 模块设计

### 5.1 Artifact Adapter 与归一化表示层

职责：

- 解析不同 artifact 输入
- 将其映射到统一中间表示
- 屏蔽不同文件格式之间的底层差异
- 保留“样式定义”和“实际生效格式”两类事实，避免被 direct formatting 覆写误导

当前 adapter：

- `.docx` adapter：解压并读取 OOXML，提取段落、run、样式、编号、节、页边距、页眉页脚、图片、表格、题注等信息；同时保留样式引用、实际生效属性和启发式角色判断依据
- PDF/text adapter：提取规范文本与段落结构

建议输出统一中间表示 `Artifact IR`。

示意字段：

```json
{
  "artifact_type": "docx",
  "ir_type": "document",
  "blocks": [],
  "styles": [],
  "layout": {},
  "metadata": {},
  "relations": []
}
```

在论文场景下，`Artifact IR` 的具体实现仍然是 `Document IR`。

### 5.2 Fact Extraction 与规则候选抽取层

`Fact Extraction` 负责从 `Artifact IR` 中提取可比事实，例如：

- 段落级事实
- 文字级事实
- 布局级事实
- 编号与结构级事实

来源分为三类：

- 模板显式样式：直接从 `.docx` 读取
- 规范文本描述：从 PDF / 文本中抽出候选规则
- 样例论文观察值：从样例中反推默认规则

该层不负责最终定论，只负责提出候选规则。

### 5.3 AI 规则归一化层

作用：

- 把自然语言规范转成 schema 内字段
- 合并重复规则
- 标记冲突和不确定性
- 识别规则适用范围

例如：

- “正文采用宋体小四，固定行距 20 磅，首行缩进 2 字符”
- 归一化为：

```json
{
  "selector": "body.paragraph",
  "properties": {
    "font_family_zh": "宋体",
    "font_size_pt": 12,
    "line_spacing_mode": "exact",
    "line_spacing_pt": 20,
    "first_line_indent_chars": 2
  },
  "source": "guideline_pdf",
  "confidence": 0.93
}
```

### 5.4 规则审核与存储层

因为模板抽取不可能百分之百准确，所以需要支持：

- 人工确认候选规则
- 规则版本化
- 规则状态管理：`draft / reviewed / published`
- 规则来源追踪

### 5.5 Comparator / Rule Engine

职责：

- 根据 selector 在 `Artifact IR` 中匹配目标对象
- 支持“规则校验”和“产物差异比较”两类执行模式
- 生成结果对象

执行对象可包括：

- 文档级：页边距、纸张、页码、节
- 区块级：封面、摘要、目录、正文、参考文献、附录
- 段落级：对齐、缩进、行距、段前后
- 文字级：字体、字号、加粗、斜体
- 表图级：题注位置、字体、编号方式、布局

建议内部分为两层：

- `rule engine`：`spec -> artifact` 合规检查
- `artifact comparator`：`artifact A -> artifact B` 差异检查

### 5.6 报告生成层

输出包含：

- 总体分数
- 按模块汇总错误
- 每条错误的定位、期望值、实际值、严重级别
- 可修复性标记

错误对象建议如下：

```json
{
  "rule_id": "body-line-spacing-001",
  "severity": "major",
  "location": {
    "section": "正文",
    "paragraph_index": 128,
    "page_hint": 14
  },
  "expected": {
    "line_spacing_pt": 20
  },
  "actual": {
    "line_spacing_pt": 18
  },
  "message": "正文第 129 段固定行距应为 20 磅，当前为 18 磅。",
  "fixable": true
}
```

## 6. 规则模型设计

规则建议采用“选择器 + 属性约束”的形式。若进入一般化比对模式，则采用“选择器 + 属性投影 + 差异计算”的形式。

### 6.1 顶层 schema

```json
{
  "spec_id": "school-thesis-v1",
  "name": "某大学本科毕业论文格式规范",
  "scope": "undergraduate_thesis",
  "version": "1.0.0",
  "artifact_type": "document",
  "document_constraints": [],
  "content_blocks": [],
  "rules": [],
  "exceptions": [],
  "sources": []
}
```

### 6.2 selector 设计

示例：

- `cover.title`
- `abstract.zh.body`
- `toc.entry.level1`
- `body.heading.level1`
- `body.paragraph`
- `figure.caption`
- `table.caption`
- `references.entry`

长期还应支持更一般的 selector 命名空间，例如：

- `block[type=paragraph]`
- `block[role=caption]`
- `layout.page.margin.top`
- `table.cell`

### 6.3 property 设计

第一批字段：

- `font_family_zh`
- `font_family_en`
- `font_size_pt`
- `bold`
- `italic`
- `alignment`
- `line_spacing_mode`
- `line_spacing_pt`
- `line_spacing_multiple`
- `space_before_pt`
- `space_after_pt`
- `first_line_indent_chars`
- `left_indent_pt`
- `right_indent_pt`
- `page_margin_top_cm`
- `page_margin_bottom_cm`
- `page_margin_left_cm`
- `page_margin_right_cm`
- `page_number_format`
- `caption_position`
- `numbering_pattern`

## 7. AI 与程序的职责分工

### 7.1 AI 负责

- 解析自然语言规范
- 从样例中概括候选规则
- 给不确定项打标签
- 生成面向用户的错误解释

### 7.2 程序负责

- 读取 artifact
- 提取结构与属性事实
- 执行规则匹配
- 输出可复核结果

### 7.3 不应让 AI 直接负责

- 最终样式值判断
- 批量逐段检查
- 数值型差异判定
- 页面级精确统计

## 8. 通用比对模式

系统至少应支持以下三种模式：

### 8.1 Compliance Check

- 输入：`spec + artifact`
- 输出：是否合规，以及违规项

### 8.2 Artifact Diff

- 输入：`artifact A + artifact B`
- 输出：两者在结构、样式、布局或语义上的差异

### 8.3 Reference Alignment

- 输入：`reference artifact + target artifact`
- 输出：目标产物与参考样例不一致的部分

## 9. 数据流设计

### 9.1 模板建模流程

1. 用户上传模板、规范、样例
2. 解析文件并生成候选事实
3. 调用模型生成候选规则 JSON
4. 对候选规则做去重、冲突检测、单位归一化
5. 输出草稿版 spec
6. 人工审核后发布

### 9.2 文档检查流程

1. 用户上传待检查 `.docx`
2. 选择 spec
3. 解析文档并生成 IR
4. 运行规则引擎
5. 聚合问题并生成报告
6. 可选地生成修复线索，但不直接改写源文件

### 9.3 通用产物比对流程

1. 用户上传 reference artifact 与 target artifact
2. 分别解析并生成 `Artifact IR`
3. 提取待比较 facts
4. 运行 comparator
5. 输出差异报告

## 10. 关键设计决策

### 10.1 为什么优先 `.docx`

- 格式语义完整
- 可以读取真实样式值
- 更适合结构化检查
- 比 PDF 更容易定位和修复

同时，`.docx` 也是建立通用抽象的好起点，因为它能迫使我们把结构、样式、布局、规则分离清楚。

但要明确一点：

- 直接解析 OOXML 不等于拿到了真实分页结果
- 页面级坐标、断页、孤行控制等问题，后续仍可能需要渲染引擎或 PDF 辅助

### 10.2 为什么先做“检查”再做“修复”

- 检查比修复更容易验证
- 修复会引入连锁副作用
- 先把规则模型打稳，后续再加自动修复成本更低

### 10.3 为什么核心概念要从第一天就抽象化

- 后续必然要支持新 artifact 类型
- “模板 vs 实际产物”的本质不只存在于论文场景
- 如果核心模块命名和接口绑定 `.docx`，后续扩展代价会很高

### 10.4 为什么规则要版本化

- 学校每年会变
- 同校不同学院会变
- 模板来源可能冲突
- 需要对历史报告可追溯

## 11. 风险与应对

### 11.1 规范文本含糊

应对：

- 允许规则置信度
- 输出待确认项
- 保留源片段引用

### 11.2 Word 文档结构不规范

应对：

- 支持“按样式识别”和“按文本模式识别”双通道
- 同时保留 `style_ref` 与实际生效格式，处理 direct formatting 覆写
- 允许对未套样式文档做启发式识别，并显式记录识别置信度

### 11.3 页面级判断难

应对：

- MVP 阶段先做“近似定位”，并允许部分规则返回 `N/A`
- 不把真实页码、跨页题注、孤行控制纳入首版硬承诺
- 后续再引入 Word 渲染或 PDF 辅助布局引擎

### 11.4 抽象过早导致复杂度膨胀

应对：

- 只抽象稳定概念，不提前实现所有 adapter
- 保持一个清晰原则：`docx-first, engine-generic`

## 12. 验收标准

- 能从真实模板中生成可用 spec
- 能检查正文、标题、摘要、目录、图表题注、参考文献等主要模块
- 输出结果可复核且解释清楚
- 接入新学校模板时，不需要重写代码
- 核心内核可容纳新的 artifact adapter，而不需要重构规则与报告模型
